# 交互与 UI 需求

## 1. 交互原则

需要聊天入口，但聊天不是产品首页，也不是唯一控制方式。桌宠本体承担“存在和发起关系”，独立终端客户端承担“深入交流、纠正、授权和追问”。

用户与 MOYU 有五种基本互动：

1. **轻触**：点击、拖动、靠近，获得即时非语言反馈。
2. **询问**：打开聊天，问它在做什么、为什么相信某件事。
3. **塑造**：纠正、允许、拒绝、要求记住或忘记。
4. **协作**：批准它继续观察、制作草稿或执行一次外部动作。
5. **探视**：查看它的新发现、收藏和未完成项目。

## 2. 聊天入口

### 打开方式

- 双击宠物或右键菜单选择“和它说话”，启动 `moyu-chat.exe` TUI。
- 可配置全局快捷键，默认不注册以避免冲突。
- 宠物气泡上的明确回复提示可进入聊天；普通气泡不抢焦点。

### 窗口形态

- 使用原生 Win32 子窗口/弹出窗口和现有软件渲染，不引入 WebView。
- 默认是紧邻宠物的小型面板，可展开但不是全屏聊天应用。
- 打开前不创建复杂控件；关闭后释放对话渲染缓存。
- 支持文本输入、滚动历史、发送、停止生成和关闭。
- 首版不要求 Markdown 富文本；仅支持纯文本、换行和少量状态标签。

### 对话语义

聊天必须支持以下明确意图：

- “你在做什么？”：解释 active intention、来源和剩余预算。
- “为什么？”：给出使用过的 belief、事件和工具，不输出伪造思维链。
- “不是这样的”：降低或删除对应 belief，并记录纠正。
- “记住……”：触发确定性 memory promotion。
- “忘掉……”：展示将删除的候选，确认后删除。
- “不要再看这个”：写入领域/路径/工具级 deny rule。
- “你可以做”：只授予当前动作或用户明确选择的持久权限。

普通闲聊允许存在，但回答应受 SOUL、MEMORY、当前情绪和关系影响，不能默认转化为工具任务。

## 3. 主动联系与收件箱

主动行为分三级：

- `silent`：只改变动作或留下收藏，不打扰。
- `ambient`：短气泡，不抢焦点，自动消失。
- `request`：需要人类决定时出现徽标/轻提示，等待用户主动打开。

禁止自主弹出聊天终端、抢键盘焦点或连续气泡轰炸。所有未处理发现和请求进入轻量“收件箱”；再次打开 TUI 时可通过状态查询查看。

## 4. 权限交互

权限请求必须说明：

- 它想做什么。
- 为什么现在想做。
- 将调用哪个工具/MCP。
- 会读取或改变什么范围。
- 这是一次授权还是持久授权。

按钮至少包含：`允许一次`、`总是允许此范围`、`拒绝`。`mutate` 动作额外显示可撤销性；不可撤销动作默认禁止，不提供“一直允许”。

用户可以从聊天或右键菜单打开权限页，撤销任何持久授权。

## 5. 桌宠本体行为

动画只能表达真实 runtime 状态，不能替代状态：

- 探索、等待授权、工具执行、获得发现、困惑、放弃、休眠。
- 同一个状态的表现可以受习惯和关系影响。
- 点击不再固定等于 happy；反应取决于当前 intention、情绪和历史互动。
- 用户拖动宠物后应持久化位置，并允许它在小范围内自主移动；不得遮挡全屏应用或持续追逐鼠标。

## 6. 可解释性界面

聊天面板提供可折叠的“为什么”摘要，而非展示内部 prompt 或 chain-of-thought：

```text
起因：发现仓库 90 天未修改
相信：这个项目可能被遗忘（置信度 0.61）
行动：只读查看最近修改时间
能力：filesystem.observe
下一步：等待相关事件，不会自动修改文件
```

用户应能从这里纠正 belief、终止 project、撤销权限或删除记忆。

## 7. 设置与诊断

右键菜单至少包含：

- 和它说话
- 最近发现
- 暂停自主行为 / 仅静默模式
- 权限与 MCP
- 打开 `~/.moyu/`
- 查看诊断
- 退出

诊断页显示 MCP 在线状态、最近一次错误、预算余量和数据库状态，不显示 secret。

## 8. UI 性能

- 保持原透明小窗口；聊天关闭时不增加持续绘制频率。
- 仅在动画、输入或事件发生时重绘；动画可使用低帧率定时器，但 idle 必须停止或降频。
- 支持 reduced motion 和完全静态模式。
- 不引入 GPU 强制依赖、大型图片解码库或 UI runtime。
