# 产品需求：一只有连续生命的桌面异宠

## 1. 核心承诺

用户第二次打开 MOYU 的原因必须是：它记得昨天，今天仍在做一件事，而且它已经因为用户的选择发生了变化。

产品必须同时提供：

- **连续性**：重启不失忆，未完成项目可以恢复。
- **自主性**：没有用户指令时也能在严格预算内形成意图。
- **可塑性**：用户的允许、拒绝、纠正、忽略会改变后续策略。
- **环境性**：接入不同 MCP 后，MOYU 的可感知世界和行为集合真正改变。
- **有限性**：它会误解、放弃、忘记，不追求全知全能。
- **后果性**：它可以留下收藏、草稿和小成果，而不只是说一句话。

## 2. 首发玩法：桌面拾荒者

MOYU 在明确授权的范围内寻找“数字遗物”，例如：

- 长期未修改的项目、被遗忘的 TODO、空目录、奇怪文件名。
- 最近失败的测试摘要、长期未合并的分支、第一条 commit。
- 日历中的异常空档、天气变化、重复出现的工作时段。
- MCP 返回的罕见或反常结果。

发现物不是通知消息，而是后续行为的种子。MOYU 可以：

- 收藏它，并给出自己的解释。
- 形成一个带置信度的错误信念。
- 连续观察数天，形成 obsession 或 project。
- 询问用户是否允许它进一步查看或制作草稿。
- 因用户纠正而改变信念和习惯。

## 3. 核心行为循环

```text
外部事件/稀疏定时器
  -> 更新 belief 和 drive
  -> Lua policy 判断是否形成 desire
  -> runtime 选择一个 intention
  -> 执行 0~3 个工具步骤
  -> 记录 episode 和结果
  -> 更新习惯、信任和情绪
  -> 必要时提升到 MEMORY.md
  -> 以动作、气泡、收藏或聊天消息表达
```

没有满足触发条件的 intention 时必须休眠，不得为了“显得活着”调用 LLM 或 MCP。

## 4. 可改变的人格

现有四个浮点人格参数不能再直接等同于人格。人格必须主要体现为策略偏好：

- 用户多次允许某类只读观察后，降低同类观察的打扰频率，但不越权升级到写操作。
- 用户持续拒绝某个领域后，减少相关 intention，并形成可见但可纠正的谨慎习惯。
- 用户经常纠正同一信念后，提高该领域的求证倾向。
- 用户长期忽略气泡后，MOYU 减少说话，更多通过收藏或聊天收件箱表达。
- 某项 MCP 长期不可用时，MOYU 记住“失去了一种感官”，而不是每秒重试。

变化必须缓慢、可解释、可重置。禁止用一次点击永久改变人格。

## 5. 项目与悬念

MOYU 同一时间最多有一个 active project。项目持续时间可跨重启和跨天，包含：

- 主题和起因。
- 当前阶段与最多三个下一步候选。
- 放弃条件和过期时间。
- 已使用的权限与工具。
- 结局：完成、误解被纠正、主动放弃、能力丢失。

示例：

> MOYU 发现一个 90 天未修改但最近被打开的仓库，误以为它“醒了”，接下来三天只在相关事件发生时观察它，最终形成收藏或询问用户。

项目不能退化为生产力任务列表，也不能替用户偷偷完成工作。

## 6. 自主行为预算

默认值，可配置：

- 每小时最多 2 次自主观察。
- 每天最多 6 次自主 MCP 调用。
- 每天最多 3 次主动打扰用户。
- 每天最多 5 次 LLM 调用；纯规则模式必须可完整运行。
- 单 intention 最多 3 次工具调用、1 次 LLM 调用、30 秒 wall time。
- 同类失败采用指数退避，进程生命周期内不得紧密重试。

用户主动聊天和明确指令可以使用独立预算，但仍受权限与总上限控制。

## 7. 明确不做

- 不做通用办公助手和无限自主 Agent。
- 不做每日签到、金币、等级、连续登录等外部奖励系统。
- 不靠随机弹话维持存在感。
- 不默认读取整个磁盘、浏览器历史、剪贴板或私人消息。
- 不伪造“我一直在后台思考”；休眠就是休眠。
- 不允许 MOYU 自行修改 `SOUL.md`。

## 8. 成功标准

封闭测试 7 天后应满足：

- 用户能说出至少一个由真实经历形成的 MOYU 习惯或误解。
- 重启后，MOYU 能正确延续最近项目和关键关系变化。
- 接入一个新 MCP 后，用户能观察到新的行为类别，而非仅多一个工具名。
- 用户可以查看“它为什么这么做”，并能纠正或禁止。
- 无 LLM 情况下仍能形成收藏、习惯、项目和可纠正信念。
